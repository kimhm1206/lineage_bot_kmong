# SQLite V2 구조

## 목표

- 반복 문장과 JSON을 저장하지 않고 숫자 ID로 관계를 연결한다.
- 아데나와 비율은 `INTEGER`로 저장한다.
- 혈맹, 유저, 수수료를 같은 정산 객체로 취급한다.
- 혈비 가계부는 수정/삭제하지 않는 공개 원장으로 관리한다.
- 현재 화면이 사용하는 기존 분배 테이블은 화면 전환이 끝날 때까지 유지한다.

## 저장 규칙

| 값 | 저장 형식 | 예시 |
| --- | --- | --- |
| 아데나 | 정수 `INTEGER` | `1250000` |
| 수수료율 | ppm 정수 | 10% = `100000` |
| 시각 | Unix seconds | `1784710800` |
| 상태 | 작은 정수 코드 | 대기 `0`, 완료 `1`, 귀속 `2` |
| 사용자/혈맹/출석/드랍 | 각 테이블 PK | `user_id`, `alliance_id` |

표시 문구는 조회할 때 ID를 조인해서 만든다. 예를 들어 감사 로그에는 문장
`드랍 등록: 블러드 투 소울 · 출석 #890 · 23:07`을 저장하지 않는다.
`action_type_id=6`, `loot_event_id`, `attendance_id`, `actor_id`, `occurred_at`만 저장하고
화면에서 아이템과 출석 정보를 조합한다.

## 감사 로그

- `audit_entity_types`: 출석, 아이템, 드랍 같은 대상 종류를 한 번만 저장한다.
- `audit_action_types`: 출석 추가, 드랍 등록 같은 작업 종류를 한 번만 저장한다.
- `audit_actors`: Discord 계정과 `user_id` 연결을 한 번만 저장한다.
- `audit_events`: 작업자, 작업 종류, 대상 ID, 시각만 저장한다.
- `audit_event_contexts`: 작업에 필요한 출석/유저/혈맹/아이템 ID만 저장한다.

기존 `work_logs.details_json`처럼 입찰 아이템 전체 상태를 매번 복제하지 않는다.
삭제된 대상은 조인이 불가능해도 `대상 #ID`로 감사 흔적이 남도록 문맥 ID에는 FK를
강제하지 않는다.

## 통합 정산 객체

- `catalog_item_versions`: 아이템 이름이 바뀔 때만 새 버전을 저장한다.
- `settlement_drops`: 출석, 아이템 버전, 원화/아데나 입력값을 정수로 저장한다.
- `settlement_drop_participants`: 드랍 당시 유저와 혈맹 ID 스냅샷이다.
- `settlement_drop_excluded_alliances`: 제외 혈맹을 JSON 대신 ID 행으로 저장한다.
- `settlement_fee_rules`: 수수료 규칙 본체다.
- `settlement_fee_rule_versions`: 이름/비율이 바뀔 때만 버전을 추가한다.
- `settlement_payout_objects`: 혈맹, 유저, 수수료를 동일한 정산 객체로 저장한다.

`object_code`는 혈맹 `1`, 유저 `2`, 수수료 `3`이다. `status_code`는 미완료 `0`,
완료 `1`, 귀속 `2`다. 상위 객체는 `parent_payout_object_id`로 연결한다.

```text
드랍 분배
├─ 경리 수수료
├─ 연합 수수료
└─ 혈맹 분배
   ├─ 혈비
   ├─ 기타 내부 수수료
   └─ 유저 분배
```

완료 버튼은 대상 객체의 상태만 바꾸며, 전체 완료는 해당 부모 아래 미완료 객체만
한 번에 갱신한다. 이미 완료/귀속된 객체는 건드리지 않는다.

## 혈비 가계부

- `treasury_accounts`: 서버-혈맹별 현재 잔액 캐시다.
- `treasury_categories`: 혈비, 귀속, 운영비 등 입출금 분류를 한 번만 저장한다.
- `treasury_source_types`: 수동, 유저 귀속, 수수료, 취소 같은 발생 원인을 코드화한다.
- `treasury_entries`: 금액, 발생 시각, 잔액, 작업자, 원본 ID만 저장하는 원장이다.

원장은 `UPDATE`와 `DELETE`가 DB 트리거로 차단된다. 잘못 입력한 내역은 원본을
지우지 않고 반대 금액의 취소 원장을 추가한다. `balance_after`는 행당 정수 하나가
더 들지만 과거 시점 잔액을 즉시 공개하고 위변조를 검증하기 위해 유지한다.

자동 귀속이나 수수료 입금은 `source_type_id + source_id` 유니크 인덱스로 중복
입금을 막는다. 사용 사유처럼 사람이 직접 입력해야 하는 내용에만 짧은 `memo`를
허용한다.

## 전환 순서

1. 기존 작업 로그를 ID형 감사 로그로 이전하고 `work_logs`를 제거한다.
2. 새 분배 등록부터 `settlement_payout_objects`를 생성한다.
3. 기존 분배 화면을 새 정산 객체 조회로 교체한다.
4. 검증이 끝난 뒤 기존 분배 상태/수수료 완료 테이블을 제거한다.
5. 혈비 귀속과 수수료 완료 시 가계부 원장을 자동 생성한다.

현재 1번과 신규 테이블 생성까지 적용되어 있다. 2~5번은 화면 개편과 함께
단계적으로 연결하며, 기존 출석 데이터는 변경하지 않는다.
